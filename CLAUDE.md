# Conductor — AI 协作规范

## 文档先行原则

**每次收到新需求，必须先写计划文档，再动手实现。**

### 判断标准

| 需求规模 | 处理方式 |
|----------|----------|
| 极小（< 5 行改动，如改文案、修 typo） | 一句话说明方案，直接执行 |
| 小（1-2 个文件，逻辑清晰） | 列出涉及文件和改动点，用户确认后执行 |
| 中大型（多文件、新功能、重构） | 在 `docs/plans/` 创建 `PLAN-<功能名>.md`，用户确认后执行 |

### PLAN 文档规范

文件命名：`docs/plans/PLAN-<kebab-case-功能名>.md`

必须包含：
1. **需求描述** — 用户想要什么
2. **方案设计** — 怎么做，关键决策
3. **涉及文件** — 哪些文件会变
4. **验收标准** — 怎么算完成

模板：`docs/plans/TEMPLATE.md`

### 执行流程

```
收到需求
  → 判断规模
  → 写 PLAN 文档（中大型需求）
  → 输出文档内容，等待用户确认
  → 用户说"确认"或"开始"后再动手
  → 实现完成后，更新文档状态为 DONE
```

### 禁止行为

- 收到需求后直接开始写代码（中大型需求）
- 跳过方案说明直接提交
- 文档和实现不一致

## 项目说明

Conductor 是本地优先的任务调度系统。详见 [docs/architecture.md](docs/architecture.md)。

AI Agent 操作 Conductor 时，先运行 `conductor help-ai` 查看命令速查表，不要凭记忆猜命令。

## 开发规范

1. Model 层无 HTTP 依赖
2. CLI 和 HTTP 共用 Model，不在 controller 里写业务逻辑
3. 类型从 `@conductor/types` 引入
4. 所有 CLI 命令支持 `--json`
5. SQLite 用 Bun 内置
6. 任务操作写 task_ops
